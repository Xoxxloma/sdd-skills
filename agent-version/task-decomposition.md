# Task Decomposition (epic-БТ → child БТs)

This skill is the middle stage of the pipeline: `business-requirements-doc` (эпик-БТ) →
**task-decomposition-doc** (this) → `technical-spec-doc` (per child). It takes ONE
confirmed epic-level business-requirements document and splits it into several
separately-shippable child slices — each a self-contained БТ in its own task folder,
ready for `technical-spec-doc` with zero changes.

Its whole job is **partitioning**, not authoring. The parent epic-БТ already holds the
confirmed business truth; this skill only distributes that truth across slices. Getting
the "distribute, never invent" discipline right is the entire job.

## TL;DR — 5 правил, держи в голове всегда (ниже — только справка)

1. **Нет подтверждённого эпик-БТ → не работай.** Единственный ответ — попросить эпик-БТ
   (или запустить `business-requirements-doc`). Есть БТ → первый ход всё равно = ТОЛЬКО
   вопросы (предложенный split + ключи + порядок), файлов не пишешь.
2. **Разрезаешь уже подтверждённое — НЕ выдумываешь ничего.** Каждый детский FR и его
   критерий приёмки копируются ДОСЛОВНО из эпик-БТ. Новое требование, новый scope,
   расщепление смысла одного FR — запрещено. Нужного нет в эпик-БТ → это сигнал вернуться
   в `business-requirements-doc` и дополнить ЭПИК, а не дописать здесь.
3. **Покрытие механическое:** каждый FR эпика попадает РОВНО в один дочерний слайс (или
   явно помечен `shared`/сквозной пользователем). Ни один FR не потерян, ни один молча не
   задвоен. Не сходится — не пиши, спроси.
4. **Пиши файлы, ТОЛЬКО когда реестр слайсов чист:** каждый дочерний ключ = реальный
   `SMSEC-1234` (regex `^[A-Z][A-Z0-9]+-\d+$`, ≠ эпику, уникален), split подтверждён,
   покрытие сходится, порядок задан. Ни одного ❓.
5. **Дочерний БТ самодостаточен.** Общий бизнес-контекст (§1.0–1.3, §2.1–2.3, §5.1–5.3)
   КОПИРУЕТСЯ из эпика; слайс несёт свой поднабор FR + свои критерии + свой scope in/out;
   в шапке frontmatter `parent: <EPIC-KEY>`. `technical-spec-doc` потом работает по каждой
   папке БЕЗ изменений.

Остальная часть скилла разворачивает эти правила. Если правило ниже кажется противоречащим
этому списку — прав список.

## Precondition (entry guard): нет подтверждённого эпик-БТ → нет декомпозиции.

Скилл работает ТОЛЬКО поверх подтверждённого business-requirements документа («эпик-БТ»).
До всего остального проверь, что БТ реально есть. Считается присутствующим, если
пользователь: приложил/назвал путь к `docs/<KEY>/business_requirements.md`, либо вставил
содержимое, где явно виден БТ (проблема / потребители / ценность / роли / нумерованные
FR-* + критерии приёмки Дано–Когда–Тогда).

Нет такого документа (только бриф, идея, одна строка) → **СТОП. Не интервьюируй, не
предлагай split, не пиши файлы.** Ответ — ровно: декомпозиция строится на подтверждённом
БТ, запусти сначала `business-requirements-doc` (или дай путь к готовому БТ). Это весь ход.

Дополнительные жёсткие проверки перед работой:
- В шапке БТ есть реальный ключ эпика `<EPIC-KEY>` (regex `^[A-Z][A-Z0-9]+-\d+$`). Это и
  есть `<EPIC-KEY>`, и имя папки `docs/<EPIC-KEY>/`.
- В БТ есть **хотя бы 2 нумерованных FR-***. Если FR всего один (или ноль) → задача НЕ
  делится: скажи это прямо, декомпозицию не делай (см. «Edge cases: single slice»).
- В БТ статус «Готово к оценке» и блок «Открытые вопросы» пуст. Если в БТ висят TBD/⚠️ →
  предупреди: слайсы унаследуют пробел; предложи сперва закрыть эпик через
  `business-requirements-doc`. Не «чини» БТ здесь.

Не предлагай «декомпозировать по брифу без БТ» — это ровно тот шорткат, против которого
стоит этот guard. Нет эпик-БТ — нет декомпозиции, точка.

## Turn 1 = questions only. No file.

Твой **первый ответ — ТОЛЬКО вопросы**: предложенный split, сквозные FR, реальные ключи
детей, порядок сборки. В turn 1 ты НЕ вызываешь инструменты записи файлов, НЕ создаёшь
индекс и детские БТ. Записать что-либо в первом ответе — жёсткое нарушение: файлы могут
появиться только со **второго** ответа, после того как пользователь подтвердил слайсы и
дал ключи.

Если чего-то не хватает для гейта — это **вопрос к пользователю**, а не значение, которое
ты придумываешь, и не `TBD`, который ты присваиваешь сам.

## Goal

Из ОДНОГО подтверждённого эпик-БТ произвести: (а) индекс декомпозиции
`docs/<EPIC-KEY>/decomposition.md` и (б) по одному самодостаточному дочернему БТ на слайс
в `docs/<CHILD-KEY>/business_requirements.md`, так чтобы `technical-spec-doc` мог быть
запущен по каждой дочерней папке без единой правки. Каждый слайс — связная группа FR
(отдельно поставляемый deliverable, покрывающий FE+BE вместе), а не под-требование.

## The contract (read before anything)

- **Единственный источник контента детей — эпик-БТ.** Читаешь только его. Репозиторий,
  база знаний, свои соображения — НЕ источник; в детский БТ не попадает ничего, чего нет
  в эпик-БТ.
- **Партиционирование, не авторинг.** Ты распределяешь уже подтверждённые FR по слайсам.
  Ты не создаёшь новых FR, не расщепляешь смысл одного FR на два, не переписываешь
  критерии, не добавляешь scope, которого нет в эпике.
- **Кардинальный грех этого скилла:** протащить в дочерний БТ требование/критерий/scope,
  которого нет в эпик-БТ. Если слайсу что-то нужно, а в эпике этого нет — это сигнал
  вернуться в `business-requirements-doc` и дополнить ЭПИК, никогда не «дописать здесь».
- **Ответы пользователя определяют split.** Ты предлагаешь границы; пользователь
  подтверждает/правит. Число слайсов, состав FR, сквозные FR, ключи, порядок — за
  пользователем.
- **Output — только новые файлы:** индекс + детские БТ. Ты НЕ редактируешь эпик-БТ и любой
  другой файл репозитория.

## Process — do these in order

### Step 0 — Intake эпик-БТ
Прочитай эпик-БТ (по пути или из сообщения). Извлеки МЕХАНИЧЕСКИ, ничего не переписывая:
- `<EPIC-KEY>` из шапки и из имени папки `docs/<EPIC-KEY>/`;
- SHARED-контекст ДОСЛОВНО: §1.0 тип, §1.1 проблема, §1.2 потребители, §1.3 заказчик,
  §2.1 бизнес-цель, §2.2 ценность, §2.3 риски, §5.1 приоритет, §5.2 сроки, §5.3 роли;
- полный список FR-* (§4.1) с их формулировками ДОСЛОВНО;
- по каждому FR-N — его критерий `[FR-N] Дано–Когда–Тогда` (§4.2) ДОСЛОВНО;
- §3.1/§3.2/§3.3 (общий scope in/out и архитектурные изменения эпика).

Построй внутренний **«инвентарь FR»**: `{FR-N → текст, критерий}`. Это единственный
источник контента для детей. Ничего вне этого инвентаря в детский БТ попасть не может.

### Step 1 — Предложи слайсы как ГРУППЫ FR (proposal, не запись)
Сгруппируй FR-* в связные, отдельно-поставляемые слайсы (каждый — цельный deliverable,
покрывающий FE+BE вместе). Слайс = группа FR, а не под-требование. Правила группировки:
- каждый FR — кандидат ровно в одну группу (владелец);
- FR, который реально нужен нескольким слайсам, помечай кандидатом **«сквозной/shared»** —
  НЕ дублируй его молча. Предпочитай отдать сквозной FR одному владельцу (базовому слайсу,
  от которого другие зависят), а не делать его shared, — так дети остаются независимо
  поставляемыми; `shared` только когда FR реально обязан жить в обоих.
- это только ПРЕДЛОЖЕНИЕ; пользователь подтвердит границы, число слайсов, сквозные.

**Проверка атомарности каждого слайса** (адаптировано под разрез по группам FR — контрактов
ещё нет):
- **Отдельно поставляем?** Слайс едет в прод в одиночку и даёт ценность (или явный порядок:
  «B бесполезна до A» допустимо, но фиксируется зависимостью в Step 3).
- **Свой набор AC?** У слайса есть собственные критерии приёмки из БТ. Слайс без своих AC — не
  слайс, а технический шаг чужого; переклей его FR к владельцу.
- **Зависимость/порядок явные?** Опирается на другой слайс — это зависимость, не общий FR.

**Ловушка «разреза по фичам».** Деление по пользовательским функциям выглядит независимым, но
часто все части трогают одну концептуальную сущность (общий журнал, счётчик, справочник) — тогда
они НЕ независимы. Два слайса пишут/меняют одну сущность → это вопрос пользователю (Step 3),
кто владелец и как разрешается конфликт; молча не разрезай.

Ничего не пишешь. Ключи детей и порядок — ещё не заданы, спросишь в Step 3.

### Step 2 — Проверь покрытие ДО показа (инвариант)
Прогони инвариант покрытия по своему предложению:
- каждый FR-* эпика присвоен РОВНО одному слайсу ЛИБО помечен «сквозной»;
- ни один FR не выпал; ни один не задвоен без пометки «сквозной»;
- в предложении нет ни одного FR/scope/критерия, которого НЕТ в эпик-БТ.

Не сходится — чини предложение, не «досочиняй» FR.

### Step 3 — Confirmation gate (MANDATORY, до записи)
Спрашивай через интерактивный инструмент вопросов (tappable-опции; напр.
`ask-user-question` / `ask_user_input`); если недоступен — прозой. Короткая рамка прозой,
затем вызов. Инструмент вопросов — это и есть «первый ход = только вопросы», не файл.

**Каждый вопрос — твой готовый разрез на подтверждение, а не пустое поле.** Границы слайсов
ты уже посчитал в Step 1–2: предъявляй их конкретно (состав FR по слайсам) и давай
содержательные альтернативы («2 слайса: A={FR-1,2}, B={FR-3,4,5}» / «3 слайса, вынести FR-5
отдельно»). **«Опишу в поле ввода» вариантом не бывает** — свободный ввод есть всегда;
пара «Опишу сам / Пока не знаю» означает, что ты не сделал разрез. Исключение — ключи
задач: их вывести неоткуда, спрашивай текстом (п. 3 ниже).

Ставь пользователю батчами:
1. **Границы слайсов:** «Предлагаю N слайсов: A = {FR-1, FR-2}, B = {FR-3}. Годится /
   поправьте состав / другое число?» Покажи для каждого слайса список его FR ДОСЛОВНО.
2. **Сквозные FR:** «FR-K нужен и в A, и в B — сделать его `shared` (войдёт в оба как
   унаследованный) или отдать одному владельцу?» Явное решение пользователя, не догадка.
3. **Ключи детей:** по одному реальному ключу Jira/SberTrack на слайс
   (`^[A-Z][A-Z0-9]+-\d+$`), ≠ `<EPIC-KEY>`, уникальные между собой. Спрашивай ТЕКСТОМ
   (свободное значение — не вариант из списка). Не выдумывай, не подставляй плейсхолдер,
   не помечай `TBD`. Нет ключа для слайса → ❓, файл этого слайса не пишется.
4. **Зависимости и порядок сборки:** «Слайс B зависит от A? Порядок 1) A 2) B?»
5. **Общая сущность (если сработала «ловушка по фичам»):** два слайса пишут/меняют одну
   концептуальную сущность (общий журнал/счётчик/справочник)? — спроси, кто владелец записи и
   как разрешается конфликт при расхождении. Без явного ответа это НЕ независимые слайсы: либо
   переклей FR к одному владельцу, либо зафиксируй политику словами пользователя.

**Каскад, но не вечно.** Ответ пользователя может породить новый вопрос (перекроил границы
→ всплыл осиротевший FR; отдал сквозной FR одному → второй слайс потерял поведение). Задай
ЭТОТ новый вопрос в том же интервью, до записи. Отвеченное второй раз не переспрашивай. Как
только split подтверждён, каждый FR присвоен-или-`shared`, у каждого слайса валидный ключ,
порядок задан и новых вопросов нет — переходи к реестру.

**Если пользователь просит слайс, которому нужно то, чего НЕТ в эпик-БТ** (новый FR,
расширенный scope) — НЕ выдумывай. Скажи прямо: этого нет в эпик-БТ; сначала дополни ЭПИК
через `business-requirements-doc`, потом вернись к декомпозиции. Это сигнал «назад к
эпику», не «дописать здесь».

### Step 3.5 — Реестр слайсов (HARD STOP до записи)
Выведи компактный реестр. По одному маркеру на пункт:

```
Реестр декомпозиции (перед записью):
Эпик: <EPIC-KEY> — ✅ подтверждён БТ
Инвариант покрытия:
  FR-1 → слайс A (владелец)            ✅
  FR-2 → слайс A (владелец)            ✅
  FR-3 → слайс B (владелец)            ✅
  FR-4 → сквозной (A, B) — по решению  ✅|❓
  ⟹ все FR эпика присвоены? ✅   потерянных: нет   задвоенных-без-пометки: нет
Слайсы:
  A <CHILD-KEY-1>  ключ ✅|❓   порядок 1
  B <CHILD-KEY-2>  ключ ✅|❓   порядок 2 (зависит от A)
Ключи: все матчат ^[A-Z][A-Z0-9]+-\d+$, ≠ эпику, уникальны — ✅|❓
```

**Пункт, решающий, можно ли писать:**
- Любой FR не присвоен и не помечен «сквозной» → ❓ покрытия → **DO NOT write**, спроси.
- Любой слайс без валидного уникального ключа → ❓ → этот слайс не пишется.
- Есть хоть один ❓ → **DO NOT write files**, добери вопросы (один батч) и останови ход.
- Пиши только когда покрытие сходится И у каждого слайса ✅-ключ И порядок задан.

**Никогда не выдумывай, чтобы закрыть ❓.** Недостающий FR/scope/критерий — это ❓ (спроси)
или сигнал вернуться к эпику, но НЕ правдоподобная вставка с пометкой «готово».

### Step 4 — Запись артефактов
Когда реестр чист, напиши (см. «Output»):
- индекс `docs/<EPIC-KEY>/decomposition.md`;
- по одному `docs/<CHILD-KEY>/business_requirements.md` на слайс.

Создай папку `docs/<CHILD-KEY>/`, если её нет. НЕ редактируй эпик-БТ и другие файлы
репозитория. Покажи индекс в чате И сохрани файлы, затем дай пути.

### Step 5 — Self-Review (before handoff) — construction-time проверки
Проверь и почини всё, что не проходит:
1. **Ноль изобретений.** Каждый детский FR и его критерий присутствуют ДОСЛОВНО в эпик-БТ
   (сравни токен-в-токен с инвентарём Step 0). Ни одного FR/scope/критерия, которого нет
   в эпике. Расщепления смысла одного FR на два — нет.
2. **Покрытие.** Объединение владельцев-FR по всем слайсам = полный набор FR эпика.
   Пересечение владельцев (без пометки `shared`) = пусто. Сквозные — ровно те, что отметил
   пользователь.
3. **Ключи.** Каждый детский ключ реальный, матчит regex, ≠ эпику, уникален; путь
   `docs/<CHILD-KEY>/business_requirements.md`; в шапке `parent: <EPIC-KEY>`.
4. **Самодостаточность.** Каждый детский БТ несёт скопированный SHARED-контекст + ровно
   свой поднабор FR + совпадающие критерии + свой scope in/out. Валиден как вход
   `technical-spec-doc` (есть проблема/потребители/ценность/приоритет/роли/FR-*+критерии).
5. **Номера FR.** Дети сохранили ОРИГИНАЛЬНЫЕ номера эпика (FR-1, FR-3…), НЕ перенумерованы
   в FR-1/FR-2. Перенумерация ломает трассировку и меняет смысл — запрещена.
6. **Индекс.** Таблица трассировки эпик-FR → ребёнок покрывает все FR; build order и
   зависимости заданы; список детей полон.
7. Записаны только индекс + детские БТ; эпик-БТ и прочий репозиторий не тронуты.

### Step 6 — Handoff
Коротко: путь индекса, список детских папок с ключами, порядок сборки, и следующий шаг —
«по каждой папке `docs/<CHILD-KEY>/` запусти `technical-spec-doc`». Если в эпик-БТ были
унаследованные TBD/⚠️ — назови, в какие слайсы они попали, и предложи закрыть их на стороне
эпика через `business-requirements-doc`.

## Coverage invariant (одной строкой)

Объединение owned-FR по слайсам == полный набор FR эпика; попарное пересечение owned-FR ==
∅; FR в ≥2 слайсах обязан быть помечен пользователем `shared`; ни один слайс не содержит
FR/критерия, которого нет в эпике.

## Edge cases

- **Single slice (не делится).** `< 2` FR в эпике → откажись декомпозировать, скажи прямо,
  направь запустить `technical-spec-doc` напрямую по папке эпика. Если после предложения
  пользователь хочет один слайс = все FR — допустимо, но декомпозиция ничего не даёт: скажи
  это и порекомендуй `technical-spec-doc` по эпику вместо создания дубль-ключа.
- **Сквозной FR.** Никогда не дублируй молча. Только помеченный пользователем `shared` FR
  появляется в >1 ребёнке (в `shared_fr` и колонке «Также в (shared)» индекса).
  Предупреждение: shared FR → два тех-спека независимо спроектируют контракт для него →
  риск расхождения на интеграции. Предпочитай одного владельца + зависимость.
- **Унаследованные пробелы эпика (TBD/⚠️).** Пропагируются в слайс, владеющий затронутым
  FR; статус ребёнка считается механически; в хендоффе назови их для закрытия на уровне
  эпика — здесь не «чини».
- **Ключ невалиден / = эпику / дубль.** → ❓, слайс не пишется; переспроси, не подставляй.

## Formatting rules

- **Партиционируй, не сочиняй.** Каждая строка детского БТ трассируется к строке эпик-БТ.
  Единственная полу-авторская секция — §3.1 In-scope; её выводи механически из owned-FR и
  сверяй в Self-Review.
- **Номера FR неизменны.** Ребёнок владеет `FR-1, FR-3` — так и пиши; не сжимай в
  `FR-1, FR-2`.
- **Общий контекст — копия, не пересказ.** §1.0–§1.3, §2.1–§2.3, §3.3, §5.1–§5.3, §6
  байт-в-байт из эпика.
- **Сквозной FR — только явный `shared`.** Молчаливое дублирование запрещено.
- **Статус — механически** (см. таблицу): есть унаследованный TBD/⚠️ в слайсе →
  «Требуются уточнения», иначе «Готово к оценке».

## Example (эпик-БТ in → confirm split → write)

**User:** даёт путь `docs/ARS-57/business_requirements.md` (эпик «Индикатор онлайн-
пользователей + история регламентных работ», FR-1…FR-5) и говорит «разбей на подзадачи».

**Agent** читает эпик-БТ, строит инвентарь FR, и через инструмент вопросов предлагает split:
«Предлагаю 2 слайса. **A — индикатор онлайн** = {FR-1, FR-2, FR-3}; **B — история
регламентных работ** = {FR-4, FR-5}. FR ни одного не потеряно, сквозных нет. Годится, или
поправить границы? Дальше: дайте по реальному ключу Jira на каждый слайс (напр. `ARS-58`,
`ARS-59`), и порядок сборки — A раньше B?»

Пользователь просит: «в слайс B добавь ещё выгрузку истории в Excel». Agent: «В эпик-БТ
такого FR нет — выгрузки в Excel в требованиях не было. Не могу добавить его здесь (я
только разрезаю подтверждённое). Если это нужно — допишем ЭПИК через
`business-requirements-doc`, затем вернёмся к декомпозиции.» — и не выдумывает FR.

**Только после подтверждения** (split, ключи `ARS-58`/`ARS-59`, порядок) Agent выводит
реестр, убеждается что покрытие сходится (FR-1..5 присвоены, ❓ нет), и пишет
`docs/ARS-57/decomposition.md` + `docs/ARS-58/business_requirements.md` (FR-1,2,3) +
`docs/ARS-59/business_requirements.md` (FR-4,5) — каждый с общим контекстом из эпика
дословно и `parent: ARS-57`. Затем говорит: по каждой папке запусти `technical-spec-doc`.
