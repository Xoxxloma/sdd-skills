## A. Модель и её характер  (перенести, если не описано)

- Движок: **Qwen3-Coder-Next** (80B MoE, 3B активных), агентный кодер.
- **Non-thinking**: модель не рассуждает «внутри». Поэтому спецификация и план — это её
  мышление, вынесенное наружу. Не проси «подумай step by step» — требуй писать подход
  явным текстом перед кодом.
- **Контекст 128K**: можно держать в контексте всю спеку + правила + нужные файлы разом,
  чанкинг не требуется.
- **Сильна в восстановлении после ошибок**: при красных гейтах вставляй ПОЛНЫЙ вывод
  ошибки обратно в контекст и итерируй точечно — НЕ переписывай файл с нуля.
- Сэмплинг (если CLI даёт настроить): рекомендация Qwen для кода — temperature 1.0,
  top_p 0.95, top_k 40. Для воспроизводимости можно снизить temperature; не наугад.

## B. Гейты: что проверять и как найти команду  (перенести, если у тебя команды захардкожены или не описаны)

Гейты заданы по смыслу, команды НЕ хардкодим — берём из проекта.

Менеджер пакетов — по lock-файлу в корне (не смешивать менеджеры):
`pnpm-lock.yaml`→pnpm · `yarn.lock`→yarn · `package-lock.json`→npm · `bun.lockb`→bun.
Запуск скрипта: `<pm> run <script>` (yarn — без `run`).

Гейты:
- **Типы (всегда):** `npx tsc --noEmit` напрямую — отдельный скрипт не нужен.
- **Линт (всегда):** скрипт `lint` из `package.json`, иначе `npx eslint .`.
- **Сборка (всегда):** скрипт `build` из `package.json`.
- **Тесты логики:** скрипт `test`, запускать неинтерактивно (для vitest — `vitest run`).

Если нужного скрипта нет и прямого инструмента тоже — не придумывай команду: отметь гейт
недоступным и сообщи человеку.

## C. Золотые правила SDD  (перенести те, которых нет — на них опираются скиллы)

1. **Сначала спецификация, потом код.** Источник истины — `specs/<feature>.requirements.md`,
   а не переписка в чате. Неоднозначно — задай вопрос, не угадывай.
2. **Одна задача за раз.** Бери один пункт из `.gigacode/plans/<feature>.plan.md`, доводи
   до зелёных гейтов, отмечай статус, только потом следующий.
3. **Проверка по типу требования.** Поведенческий R-id закрывается тестом своего
   хука/функции (ссылка на R-id в тесте). Визуальный/разметочный R-id — пунктом ручного
   чек-листа в плане.
4. **Логика вне компонента.** Поведение (дебаунс, гонки, валидация, переходы состояний,
   навигация) живёт в хуках/чистых функциях; компонент — тонкая оболочка. Если тест
   требует тяжёлого мокинга — не мокай, вынеси логику в хук и тестируй там.
5. *(опц.)* Без новых зависимостей без явного запроса; не трогать файлы вне области задачи.

## D. Definition of Done  (перенести/смержить со своим чек-листом)

- [ ] Реализованы только пункты текущей задачи из плана.
- [ ] Активные гейты зелёные (типы `tsc --noEmit`, линт/сборка/тесты — из `package.json`);
      тесты логики зелёные.
- [ ] Поведенческие R-id покрыты тестами хуков/функций; визуальные — пройдены по ручному
      чек-листу и перечислены в ответе как «проверить вручную: …».
- [ ] Статус задачи в `.gigacode/plans/<feature>.plan.md` обновлён; диф минимальный.
