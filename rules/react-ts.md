# Правила: React + TypeScript (frontend)

> Детализация к `GIGACODE.md`. Каждое правило конкретно и проверяемо.

## Структура
- Код фичи: `src/features/<feature>/`. Внутри — `components/`, `hooks/`, `logic.ts`
  (чистые функции/редьюсеры), `api/`, `types.ts`, `__tests__/`.
  Переиспользуемые примитивы UI: `src/shared/ui/`.
- Один компонент = один файл `PascalCase.tsx`. Хук = `useXxx.ts` + `useXxx.test.ts` рядом.
- Барель-файлы (`index.ts`) только на границе фичи.

## Архитектура: логика вне компонентов (главное правило тестируемости)
- Бизнес-логика и поведение НЕ живут в теле компонента. Выноси:
  - в кастомные хуки `useXxx` — поведение с состоянием/эффектами (дебаунс, запросы +
    отмена, навигация по списку, фокус);
  - в чистые функции и редьюсеры в `logic.ts` — преобразования, валидация, переходы
    состояний (без React, без DOM).
- Компонент = тонкая оболочка: вызвать хук(и), отрендерить результат, проставить ARIA,
  пробросить коллбэки. В компоненте НЕТ `fetch`, НЕТ таймеров, НЕТ бизнес-условий
  сложнее тернарника для отображения. Причина: только так поведение тестируется без
  монтирования компонента и без горы моков.
- Эвристика мок-бюджета: если для теста нужно мокать больше 1–2 зависимостей или
  монтировать «пол-приложения» — логика лежит не там. Не пиши тяжёлый тест — вынеси
  логику в хук/функцию и протестируй её изолированно.

## Компоненты
- Только функциональные компоненты и хуки. Классовые не использовать.
- Пропсы — явный `type Props = { ... }`; никаких `React.FC`.
- Компонент не делает сетевых запросов в теле — данные через хук или пропсы.
- Эффекты — в `useEffect`/`useCallback` с корректным массивом зависимостей.
  `react-hooks/exhaustive-deps` не отключать.
- Списки — со стабильным `key` по id сущности, не по индексу.

## TypeScript
- `strict: true`. Запрещены `any`, `as any`, `@ts-ignore`, `@ts-expect-error` без причины.
- Внешние/непроверенные данные приходят как `unknown` и сужаются перед использованием.
- Состояния — дискриминируемые юнионы:
  `type State = { status:'idle' } | { status:'loading' } |
   { status:'success'; data:T } | { status:'error'; error:string }`.
  Причина: исключает «невозможные» состояния на типах. Такой `State` — естественная
  единица для тестирования логики в хуке/редьюсере.

## Данные
- Запросы: `fetch` с `AbortController`; отменять при размонтировании и при старте нового
  запроса (защита от гонок). Вся эта механика — внутри хука, не в компоненте.
- Ошибки сети не глотать: хук отдаёт состояние ошибки, компонент его рендерит.

## Доступность (a11y)
- Интерактив доступен с клавиатуры (Tab/Enter/Esc/стрелки по контексту).
- Кастомные виджеты следуют WAI-ARIA: корректные `role`, `aria-expanded`,
  `aria-controls`, `aria-activedescendant`.
- Плейсхолдер дублируется доступной меткой (`aria-label`/`<label>`).

## Тестирование (политика гейтов)
Гейты `typecheck` / `lint` / `build` — всегда. Про тесты:
- **Тестируем (обязательно):** хуки (`renderHook` + fake timers где нужно), чистые
  функции и редьюсеры из `logic.ts`. Здесь поведение, которое глазами не проверишь:
  дебаунс, гонки, валидация, переходы состояний, навигация по списку.
- **Не тестируем юнит-тестами:** чистый рендер и визуальное оформление — проверяется
  в ревью/в запущенном приложении.
- **Исключение:** если поведение проявляется ТОЛЬКО на уровне компонента и рискованно
  (например, перевод фокуса), допустим один точечный render-тест. Это исключение, не норма.
- Поведенческий R-id закрывается тестом своего хука/функции. R-id про разметку/визуал —
  пунктом ручного чек-листа приёмки (см. план фичи).
- Тесты — по ролям/тексту (`getByRole`/`getByText`), асинхронность через
  `findBy…`/`waitFor`, без произвольных `setTimeout`. В начале теста комментарий с R-id.
