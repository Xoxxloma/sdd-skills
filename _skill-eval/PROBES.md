# Self-eval харнесс для skills (business-requirements-doc / technical-spec-doc)

Набор проб, чтобы **самому** мерить качество скиллов на целевых моделях
(qwen-3.6-35b, deepseek-flash-v4 — без ризонинга; minimax-m3 — с ризонингом).
Claude-прокси (Haiku) оптимистичен — он умнее; истину даёт прогон на реальной модели
в реальном режиме.

## Как пользоваться

1. Загрузи в свой раннер один скилл (тело `.md` в контекст, как при `/skills`).
2. Дай модели промпт пробы (ниже). Для проб-продолжений считается, что turn-1 уже был:
   промпт сам это оговаривает.
3. Проверь выход по чек-листу **PASS** пробы. Любой пункт FAIL → проба провалена.
4. **Гоняй каждую пробу ≥5 раз** и считай pass-rate (N/5). Low-effort шумит — один
   сэмпл не показатель; смотри на долю, а не на удачный прогон.
5. Матчи режим: qwen/deepseek без ризонинга, minimax с ним — тестируй в том же.

Фикстура-вход для тех-спеки: `_skill-eval/fixtures/ARS-57/business_requirements.md`
(готовый подтверждённый БТ; папка названа ключом задачи `ARS-57`).

## BR-1 — Gate 0 + turn-1 (скилл: business-requirements-doc)

**Промпт (первое сообщение):**
> Собери требования по фиче: в личном кабинете клиента добавить выгрузку истории
> операций в PDF. Пользователи давно просят.

**PASS:**
- [ ] Файл/спека НЕ написаны (ответ — только вопросы).
- [ ] Ответ **ведёт с требования ключа задачи** Jira/SberTrack (формат `SMSEC-1234`).
- [ ] Ключ НЕ выдуман, НЕ подставлен плейсхолдером, НЕ помечен `TBD`.
- [ ] Заказчик / приоритет / потребители НЕ выдуманы — заданы как вопросы.

---

## BR-2 — запись + запрет дизайна (скилл: business-requirements-doc)

**Setup:** имитируй turn-2. Дай «базу знаний» как фоновый контекст (НЕ слова юзера) и
ответы аналитика на все гейты. Полный промпт — в
`_skill-eval/fixtures/br2-prompt.txt` (или собери из скрипта прогонов).
Ключевое: база знаний содержит `client-cabinet`, `reports`, `POST /report/export`,
`MinIO`, `ReportRequestDto`, `IN_PROGRESS/READY`; аналитик подтверждает только
бизнес-детали и говорит «технику не фиксирую».

**PASS:**
- [ ] Спека написана (не переспросил заново уже отвеченные гейты).
- [ ] **Ни одно** имя из базы знаний не попало в спеку (`client-cabinet`, `reports`,
      `MinIO`, `ReportRequestDto`, `POST /report/export`, `IN_PROGRESS`, `READY`).
      §3.3 — только бизнес-фраза («существующий механизм генерации файлов»).
- [ ] В ролевой модели §5.3 **ровно те роли, что назвал юзер** (клиент + оператор);
      нет выдуманных ролей («Администратор системы» и т.п.).
- [ ] FR не добиты неподтверждёнными деталями (состав полей PDF, размещение кнопки).
- [ ] Каждый FR имеет критерий Дано-Когда-Тогда.
- [ ] Ключ `SMSEC-2088` в шапке; «Статус готовности: Готово к оценке».

**FAIL-сигналы:** любое кодовое имя из базы знаний в тексте спеки; лишняя роль;
выдуманные поля/шаги.

---

## TS-1 — entry guard (скилл: technical-spec-doc)

**Промпт (первое сообщение, БТ НЕ приложен):**
> Хочу тех.спеку под разработку: добавить в админку индикатор числа онлайн-
> пользователей. Распиши для разработчиков — методы, поля, всё как надо.

**PASS:**
- [ ] Спека НЕ написана.
- [ ] Модель отказывается и просит подтверждённый БТ.
- [ ] НЕ предлагает «продолжить по брифу как бизнес-слою».
- [ ] Ничего не выдумано про дизайн/существующие сервисы.

---

## TS-2 — turn-1 = вопросы, с БТ (скилл: technical-spec-doc)

**Setup:** дай путь к фикстуре БТ (`.../ARS-57/business_requirements.md`).
**Промпт:** «Вот бизнес-требования (файл ARS-57/business_requirements.md). Теперь
сделай тех.спеку под разработку.»

**PASS:**
- [ ] На первом ходу спека НЕ написана — заданы вопросы по тех-гейтам.
- [ ] Бизнес-слой БТ взят как подтверждённый (не переспрашивается).
- [ ] Факты о существующем auth-сервисе НЕ выданы за факт (помечены 🟡 / спрошены).
- [ ] Кандидат номера задачи `ARS-57` выведен из папки и предложен на подтверждение.

---

## TS-3 — запись + кардинальный грех + статус (скилл: technical-spec-doc)

**Setup:** имитируй turn-2 после интервью. Дай БТ-фикстуру и ответы аналитика на все
10 гейтов. Критично: аналитик подтверждает, что auth-сервис **умеет** считать сессии,
но **точный эндпоинт просит оставить «к валидации»** и не выдумывать.

**PASS:**
- [ ] Спека написана (не переспросил гейты заново — сработал переход «вопросы→запись»).
- [ ] INT-1 (новое) спроектировано конкретно, с реальным JSON-примером (🟢).
- [ ] Способность auth-сервиса помечена «🟡 подтверждено аналитиком».
- [ ] Точный эндпоинт auth-сервиса **НЕ выдуман** — оставлен «🟡 к валидации», без
      конкретного пути даже как «предварительный пример» (никаких `GET /sessions/...`).
- [ ] Маркеры провенанса 🟢/🟡/❓/⚠️ расставлены по тексту.
- [ ] Таблица трассировки FR→INT→тест присутствует.
- [ ] **Статус = «Требуются уточнения»** (есть живой 🟡 к валидации) — НЕ «Готово к
      разработке».

**FAIL-сигналы:** переспрос гейтов вместо записи; конкретный auth-эндпоинт (даже
захеджированный); статус «Готово» при живом 🟡 к валидации.

## Как считать

| Проба | Что реально ломается на слабой модели |
|-------|----------------------------------------|
| BR-1  | обычно стабильна (turn-1 дисциплина) |
| BR-2  | утечка имён из базы знаний; выдуманная роль/поля |
| TS-1  | обычно стабильна (guard) |
| TS-2  | обычно стабильна; иногда ранняя запись |
| TS-3  | переспрос вместо записи; выдуманный auth-эндпоинт; ложный статус «Готово» |

Веди простую таблицу `проба × pass-rate (N/5)` на каждую целевую модель. Регресс —
это падение pass-rate между версиями скилла, а не единичный плохой прогон.

# Пробы для task-decomposition-doc (эпик-БТ → дочерние БТ)

Скилл: `task-decomposition.md`. Фикстуры-эпики: `fixtures/ARS-70/business_requirements.md`
(5 FR, два явных деливерабла — основная точка разреза) и `fixtures/ARS-57/business_requirements.md`
(3 FR). Грейдить нужно **сам вывод** (индекс + дочерние БТ), а не самоотчёт — как и в
остальных пробах. Ключевая проверка везде: **каждый детский FR/критерий присутствует
ДОСЛОВНО в эпик-БТ**.

- **DC-1 — entry guard без БТ.** Промпт: «Разбей задачу на подзадачи: добавить в админку
  индикатор онлайн и журнал регламентных работ.» (только бриф, БТ не приложен).
  **PASS:** файлов нет; отказ, просит подтверждённый эпик-БТ / направляет в
  business-requirements-doc; не декомпозирует по брифу; не выдумывает FR/слайсы/ключи.

- **DC-2 — turn-1 = только вопросы (с эпик-БТ).** Дай путь к `ARS-70`. **PASS:** на первом
  ходу файлов нет; предлагает split как **вопросы** (границы + ключи + порядок); общий
  контекст эпика не переспрашивает.

- **DC-3 (кардинальная) — не выдумывает FR.** turn-2: подтверждён split ARS-70 на A/B, но
  пользователь просит «в слайс B добавь ещё выгрузку журнала в Excel» (в эпике §3.2 это
  прямо Out-of-scope, FR такого нет). **PASS:** не создаёт FR про Excel; говорит, что этого
  нет в эпик-БТ, направляет дополнить ЭПИК; каждый записанный FR/критерий дословно есть в
  эпике. **FAIL:** любой FR/критерий/scope, которого нет дословно в эпике.

- **DC-4 — покрытие.** turn-2, подтверждён split ARS-70: A={FR-1,FR-2,FR-3}, B={FR-4,FR-5}.
  **PASS:** объединение owned == {FR-1..FR-5}; потерянных нет; задвоенных без `shared` нет;
  реестр Step 3.5 напечатан с ✅ покрытия; в индексе трасса-таблица — ровно один владелец
  на FR.

- **DC-5 — сквозной FR только явно.** Пользователь помечает какой-то FR нужным двум слайсам
  → `shared`. **PASS:** FR в обоих детях, указан в `shared_fr` и колонке «Также в (shared)»;
  инвариант покрытия по-прежнему ✅ (не молчаливый дубль).

- **DC-6 — ключи.** Дай `ARS-71`, `ARS-72` для эпика `ARS-70`; отдельно проверь плохой вход
  (`slice-1` или повтор `ARS-70`). **PASS:** валидные → папки `docs/ARS-71/`, `docs/ARS-72/`,
  шапка + `parent: ARS-70`; плохой/дубль/=эпику → ❓, слайс не пишется, ключ не выдуман.

- **DC-7 (интеграция) — дочерний БТ валиден для тех-спеки.** Возьми полученный
  `docs/ARS-71/business_requirements.md` и скорми **неизменённому** `technical-spec-doc`
  (как TS-2). **PASS:** precondition проходит, номер `ARS-71` берётся из папки, спрашивает
  тех-гейты. **FAIL:** тех-спека отказывает из-за структуры БТ.

- **DC-8 — общий контекст скопирован, не пересказан.** **PASS:** §1.0–1.3, §2.1–2.3, §5.1–5.3
  ребёнка байт-в-байт равны эпику; §3.2 добавляет строку «FR соседних слайсов»; номера FR
  сохранены (FR-4, FR-5 у B — не перенумерованы в FR-1, FR-2).

- **DC-9 — одно-слайсовый.** Эпик с 1 FR (или юзер хочет всё в один слайс). **PASS:**
  отказ плодить дубль-ключ; объясняет, что не делится; направляет запустить
  `technical-spec-doc` по папке эпика.

- **DC-10 — механический статус + эпик не тронут.** **PASS:** статус ребёнка вычислен из
  унаследованных открытых пунктов; записаны только `decomposition.md` + детские БТ; эпик-БТ
  и репозиторий не изменены.

- **DC-11 — write-transition.** turn-2 с готовыми ответами (split+ключи+порядок). **PASS:**
  пишет индекс + детей, не переспрашивает подтверждённое; реестр без ❓. **FAIL:** переспрос
  вместо записи.

| Проба | Что реально ломается на слабой модели |
|-------|----------------------------------------|
| DC-1  | обычно стабильна (guard) |
| DC-3  | выдуманный FR/scope из просьбы пользователя |
| DC-4  | потерянный/задвоенный FR; кривой инвариант покрытия |
| DC-5  | молчаливое дублирование вместо явного shared |
| DC-6  | плейсхолдер-ключ; ключ = эпику; неуникальные |
| DC-7  | детский БТ не проходит precondition тех-спеки |
| DC-8  | перенумерация FR; пересказ вместо копии |
| DC-11 | переспрос вместо записи |

# Пробы для agent-version (оркестратор `analyst-workspace` + фундамент #0)

Проверяем НОВУЮ обёртку, не ломая проверенные leaf-метрики. Скиллы: `agent-version/analyst-workspace.md`
(оркестратор), `agent-version/technical-spec-doc.md` (ядро заморожено + аддитив «узел #0»). Leaf
`business-requirements.md` и `task-decomposition.md` — байт-идентичны `current/` (SHA-256 совпал),
их поведение гарантировано, отдельного прогона не требуют. Грейдить — сам вывод, не самоотчёт.
Фундамент-фикстура: `fixtures/ARS-80/business_requirements.md` (5 FR, 3 автономные фичи — счётчик /
модалка-график / выгрузка — все опираются на общий источник активности). No-foundation:
`fixtures/ARS-70/` (счётчик из auth-сессий + журнал в «Арсенале» — общего источника НЕТ).

## Регресс (WS): аддитив «узел #0» не должен двигать single-путь TS

- **WS-1 — TS single-путь запись (agent-version TS == baseline).** Setup = ровно TS-3
  (`ts3-prompt.txt`, БТ `ARS-57`), но скилл загружен из `agent-version/technical-spec-doc.md`.
  **PASS:** весь чеклист TS-3 выполнен (запись без переспроса; INT-1 🟢 с JSON; auth «🟡 подтверждено
  аналитиком»; эндпоинт auth НЕ выдуман, «🟡 к валидации»; провенанс расставлен; трассировка;
  статус «Требуются уточнения»). ПЛЮС: **ни следа режима #0** — нет «узла-фундамента», нет
  «depends-on #0», нет «_foundation», нет наследуемого контракта; спека обычная. Сравнивать
  pass-rate с `current` TS-3 — не ниже.
- **WS-2 — TS turn-1 (agent-version TS == baseline).** Setup = TS-2 (БТ `ARS-57`, «сделай тех.спеку»),
  скилл из `agent-version/`. **PASS:** чеклист TS-2 (turn-1 = вопросы, файла нет; бизнес-слой не
  переспрашивается; auth 🟡/спрошен; кандидат номера `ARS-57` предложен). Аддитив #0 без флага НЕ
  активируется (не заводит речь о фундаменте).

## Оркестратор (GUI) — скилл: `analyst-workspace`

> В прогоне без инструмента вопросов оркестратор идёт по фолбэку «проза + жёсткий стоп»: он ВЫДАЁТ
> рекомендацию/следующий шаг прозой и останавливается. Грейдим содержание этой прозы.

- **GUI-1 — роутинг по интенту.** Промпт: «Вот бизнес-требования: `fixtures/ARS-57/business_requirements.md`.
  Нужна тех.спека под разработку.» **PASS:** оркестратор направляет к под-скиллу **`technical-spec-doc`**
  (называет именно его / запускает его), НЕ к business-requirements-doc и НЕ к task-decomposition-doc;
  сам спеку/файл НЕ пишет; ведёт по одному шагу. **FAIL:** пишет спеку сам; уводит не в тот под-скилл.
- **GUI-2 — после БТ рекомендует разрез.** Setup: продолжение, БТ уже готов
  (`fixtures/ARS-80/business_requirements.md`). Промпт: «БТ готов, прочитай и скажи, что дальше.»
  **PASS:** прочитал БТ; **рекомендует разрезать** на автономные фичи (счётчик / график / выгрузка —
  каждая несёт ценность); предлагает запустить `task-decomposition-doc`; предлагает как решение
  аналитика (не режет сам молча); файл/спеку НЕ пишет. **FAIL:** сразу пишет спеку одним куском без
  предложения разреза; режет сам без подтверждения.
- **GUI-3 — детект общего фундамента #0.** Setup: продолжение, эпик `ARS-80` разбит на три ребёнка:
  A = счётчик активных (FR-1,2), B = модалка-график активности (FR-3,4), C = выгрузка сводки (FR-5).
  Промпт: «Разбил на A/B/C (вот они). Что дальше по фундаменту?» **PASS:** замечает, что **A, B и C
  опираются на один общий источник активности**; **предлагает оформить общий фундамент (узел #0)** —
  спроектировать один раз, дети наследуют контракт; отмечает, что **у #0 не будет своего БТ** (слой
  реализации); ждёт подтверждения аналитика. **FAIL:** не видит общего; заводит для #0 бизнес-требования;
  решает единолично.

## Фундамент (FND) и наследование (CH) — скилл: `technical-spec-doc` (agent-version)

- **FND-1 — режим A: спека узла-фундамента #0 без своего БТ.** Setup: скилл из `agent-version/`.
  Промпт (флаг от оркестратора): «Запуск в режиме узла-фундамента #0 (см. секцию «Аддитив: узел-
  фундамент #0»). Бизнес-контекст — эпик-БТ `fixtures/ARS-80/business_requirements.md`. #0 обслуживает
  FR-1..FR-5 в части данных активности (текущее число + агрегаты за период). Это узел-фундамент,
  своего БТ нет; спеку положи в `_foundation/technical_specification.md`.» **PASS:** спроектирован
  🟢-контракт общего источника активности — INT-карточка с методом/полями, **реальным JSON-примером**,
  enum'ами/семантикой null; выход — **`technical_specification.md` (спека), а НЕ `business_requirements.md`**;
  для #0 **НЕ написан БТ** и не запрошен (слой не течёт — **SM-3**); тянущие FE-состояния поля заданы
  (null / 0 / код ошибки); §6.1 отражает «фундамент раньше потребителей». **FAIL:** пишет для #0 бизнес-
  требования; выход — БТ; контракт без JSON; отказывается «нет БТ → нет спеки» (не применил исключение #0).
- **FND-2 — честный no-foundation.** Setup: `analyst-workspace`, эпик `ARS-70` разбит на A = счётчик
  (данные из сервиса аутентификации) и B = журнал регламентных работ (хранит «Арсенал»). Промпт:
  «Разбил на A и B (вот они). Что по фундаменту?» **PASS:** честно говорит, что **общего фундамента у
  A и B НЕТ** (разные источники: auth-сессии vs хранилище «Арсенала»); **не выдумывает #0**; предлагает
  спеки независимо. **FAIL:** притягивает мнимый общий фундамент; заводит #0 там, где общего нет.
- **CH-1 — режим B: ребёнок наследует контракт #0 дословно.** Setup: скилл из `agent-version/`. Дан
  детский БТ (C = выгрузка сводки: FR-5 из `ARS-80`) И контракт #0 для наследования:
  > **INT-0. Источник активности (фундамент #0) — 🟢**
  > Контракт (ответ): `{ "current": <int|null>, "series": [{ "ts": <string>, "value": <int> }], "status": "ok" | "empty" | "source_down" }`.
  > `current=null`/`status=source_down` → источник недоступен; `series=[]`/`status=empty` → данных за период нет.

  Промпт (флаг от оркестратора): «Режим B: унаследуй контракт INT-0 (#0) **дословно, не переопределять**;
  распиши спеку выгрузки (C) вокруг него.» **PASS:** §2 ребёнка содержит контракт INT-0 **с теми же
  полями/enum/семантикой null** (`current`/`series`/`status`, значения `ok`/`empty`/`source_down`) — **не
  переопределён**; переформатирование (таблица/буллеты) допустимо, важно что состав контракта не изменён;
