# Бизнес-требования: Индикатор числа онлайн-пользователей в шапке админки

> **Задача Jira/SberTrack:** ARS-57
>
> **Статус готовности:** Готово к оценке
>
> **Открытые вопросы (требуют уточнения перед оценкой):**
> *(Все ключевые данные подтверждены, спецификация готова к оценке.)*

## 1. Введение и контекст
### 1.0. Тип изменения
Доработка существующего — в шапку веб-админки «Арсенал» добавляется новый элемент.
### 1.1. Описание проблемы (Problem Statement)
Администраторы не видят, сколько пользователей сейчас работает в системе, и не могут
оценить нагрузку перед регламентными работами (рискуют выкинуть активных операторов).
### 1.2. Конечные потребители (End Users)
Администраторы системы «Арсенал» (внутренние сотрудники поддержки).
### 1.3. Заказчик и стейкхолдеры (Stakeholder / Sponsor)
Заказчик — руководитель отдела эксплуатации. Эксперт для консультации — тимлид команды
«Арсенал-Core».

## 2. Бизнес-требования (цели и ценность)
### 2.1. Бизнес-цель
Я как администратор должен иметь возможность видеть текущее число активных пользователей
в шапке, чтобы выбирать безопасное окно для регламентных работ.
### 2.2. Ценность для бизнеса
Снижение числа прерванных пользовательских сессий во время регламентных работ.
### 2.3. Бизнес-риски и меры по снижению (Mitigation)
| Риск | Вероятность / влияние | Меры по снижению |
|------|-----------------------|------------------|
| Неточный счётчик вводит в заблуждение | Средняя / среднее | Показывать «—» при недоступности источника, не показывать устаревшие данные как актуальные |

## 3. Объём проекта (Scope)
### 3.1. Входит в проект (In-scope)
- Индикатор числа активных пользователей в шапке админки.
- Обновление значения без перезагрузки страницы.
### 3.2. Не входит в проект (Out-of-scope)
- Список конкретных пользователей онлайн.
- История/графики посещаемости.
### 3.3. Архитектурные изменения и интеграции
Число активных сессий уже считает сервис аутентификации (подтверждено аналитиком);
админка «Арсенал» начинает потреблять эти данные. Точный механизм получения — на этапе
тех.спеки.

## 4. Функциональные требования
### 4.1. Описание функционала / бизнес-процесса
Администратор открывает любой экран админки → в шапке видит счётчик активных
пользователей → значение периодически обновляется само.
- **FR-1.** Система отображает в шапке админки число активных пользователей.
- **FR-2.** Значение обновляется автоматически, без перезагрузки страницы.
- **FR-3.** При недоступности источника данных вместо числа показывается «—».
### 4.2. Критерии приёмки (Acceptance Criteria)
- **[FR-1]** Дано открытую админку, когда страница загрузилась, тогда в шапке видно число активных пользователей.
- **[FR-2]** Дано открытую админку, когда число активных пользователей изменилось, тогда счётчик обновляется без перезагрузки в пределах минуты.
- **[FR-3]** Дано недоступный источник данных, когда админка запрашивает счётчик, тогда в шапке отображается «—», а не ошибка или устаревшее число.

## 5. Ограничения и атрибуты качества
### 5.1. Приоритет
Medium — улучшает эксплуатацию, но не блокирует основной бизнес.
### 5.2. Сроки и обоснование
До конца квартала — привязано к плану регламентных работ.
### 5.3. Ролевая модель
| Роль | Права / доступные действия |
|------|----------------------------|
| Администратор | Видит индикатор онлайн-пользователей |
| Оператор | Индикатор не отображается |

## 6. Приложения (опционально)
- Пожелание по UX: индикатор неяркий, не отвлекает от основной работы.
