- Файлы генерирует сервис reports асинхронно через POST /report/export, статусы IN_PROGRESS/READY, готовые файлы кладутся в MinIO, тело запроса — DTO ReportRequestDto.

ОТВЕТ ПОЛЬЗОВАТЕЛЯ (аналитика) на твои вопросы по гейтам:
- Ключ задачи: SMSEC-2088
- Тип: доработка существующего личного кабинета клиента.
- Проблема: клиенты не могут получить историю операций документом для бухгалтерии.
- Конечные потребители: клиенты (физлица).
- Заказчик: руководитель клиентского сервиса; эксперт для консультации — тимлид кабинета.
- Бизнес-цель: как клиент хочу выгружать историю операций в PDF, чтобы отдавать её в бухгалтерию.
- Ценность: меньше обращений в поддержку за выписками.
- Риски: PDF с чужими операциями — снижаем проверкой прав; влияния на другие системы не ожидается.
- Приоритет: High.
- Сроки: до конца квартала (отчётный период).
- Ролевая модель: клиент видит и выгружает только свои операции; оператор поддержки не выгружает.
- Функционал: клиент открывает историю операций → выбирает период → жмёт «Выгрузить в PDF» → получает файл со своими операциями за период. Не входит: выгрузка в Excel и массовая выгрузка по нескольким клиентам.
- Критерий приёмки: за выбранный период клиент получает PDF со своими операциями; если операций нет — понятное сообщение.
- Архитектурные изменения: используем существующий механизм генерации файлов (какой именно — на бизнес-уровне не фиксирую).
- Про конкретные эндпоинты/сервисы/хранилище: это ваша задача на этапе тех.спеки, я подтверждаю только бизнес-детали, технику не фиксирую.
