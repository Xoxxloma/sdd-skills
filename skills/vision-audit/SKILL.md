Стратегический аудит продукта **сверху вниз**: насколько система соответствует своему видению, какие SDLC-исходы — разрывы, и какие из разрывов стоит брать (с учётом веб-исследования, конкурентов и собственных принципов продукта). Опциональная стратегическая стадия конвейера: запускается после `system-map`, питает тот же бэклог, что и `propose-improvements` (только сверху вниз, а не снизу вверх от smells/линз).

**Ключевой принцип:** мыслить **исходами** (outcomes), а не практиками/инструментами. Возможность ценна, только если приближает видение, проходит фильтр принципов продукта и даёт дифференциацию от конкурентов.

## Inputs / outputs
- Читает (входы видения — **опционально**): `product/vision/VISION.md`, `product/vision/SDLC_REFERENCE.md`, `product/vision/COMPETITORS.md`.
- Читает (доказательная база — **обязательно**): `product/system/system-map.md` + `product/system/services/*.md` + `product/manifest.yaml`. Это источник «что система делает» — **не** грепать исходники напрямую. Если `system-map.md` нет → стоп: «сначала запусти `/system-map`».
- Может читать веб через `WebSearch` (если не указан `--no-web`).
- Пишет: `product/vision/vision-audit.md` (единственный выход — Part A/B + список `OPP-NNN` + таблица OPP→IMP).
- Шаблоны входных документов: в `reference/` этого скилла (`VISION.template.md`, `SDLC_REFERENCE.template.md`, `COMPETITORS.template.md`); шаблон самого отчёта — `reference/vision-audit.template.md`.
- **Read-only на исходники сервисов.** Пишет только в `product/vision/`.

## Режимы
- **Обычный** — документы видения заполнены в `product/vision/`. Оценки Part A имеют полный вес («видение авторское»).
- **`--lite`** — документов видения нет. Скилл **выводит неявное видение** (цель, кандидаты-принципы, scope) из карты системы и **подтверждает у пользователя** перед оценкой. В отчёте видение помечается «выведено и подтверждено» — оценки несут меньший вес.
- **`--no-web`** — без веб-исследования. Part C опирается только на `COMPETITORS.md` (если есть) и карту. Уместно для закрытого контура.

## Procedure

### 1. Разобрать документы видения
Прочитать `VISION.md` (проблема, цель-aspiration, **нумерованные принципы** в порядке приоритета, сильные/слабые стороны, scope in/out, измеримые критерии успеха) и `SDLC_REFERENCE.md` (исходы по фазам с id `1.1, 1.2…` и «почему важно»).
- Если документов нет: предложить шаблоны из `reference/` (скопировать в `product/vision/` и попросить заполнить) **или** перейти в `--lite` с подтверждением выведенного видения. **Не выдумывать видение молча** (паттерн `system-map`).
- `COMPETITORS.md` нет — продолжать, отметив «без конкурентного контекста».

### 2. Собрать доказательства из карты системы
Из `system-map.md` (каталог возможностей, сквозные сценарии, cross-cutting наблюдения) и карточек сервисов (Key flows, Risks/smells, UI surfaces) собрать факты о том, что система реально делает и где у неё дыры. Каждую дальнейшую оценку якорить на конкретику отсюда (возможность, поток, сервис, риск).

### 3. Part A — оценка соответствия видению (R/Y/G)
По каждому пункту видения (цель, каждый принцип, сильные/слабые стороны, scope, критерии успеха) выставить оценку с доказательством из шага 2:

| Оценка | Значение |
|---|---|
| 🟢 Green | Соответствует — есть явное подтверждение |
| 🟡 Yellow | Частично — смешанные сигналы |
| 🔴 Red | Противоречит — есть контрпример |

### 4. Part B — классификация SDLC-исходов
Для каждого исхода из `SDLC_REFERENCE.md` (или, в `--lite`, из стандартного продуктового SDLC-набора шаблона) — одно из трёх:
```
ACHIEVED     → система достигает исхода (есть подтверждение в карте)
AVOIDED      → вне scope по VISION.md (с цитатой исключения)
OPPORTUNITY  → в scope, но ещё не достигнут → идёт в Part C
```
Дерево решений:
```
Система достигает исхода?
├─ ДА → ACHIEVED
├─ НЕТ → Вне scope по VISION.md?
│        ├─ ДА → AVOIDED (процитировать исключение из scope)
│        └─ НЕТ → OPPORTUNITY (на исследование)
```

### 5. Part C — исследование и фильтр возможностей
Для каждой OPPORTUNITY:
- **Веб-исследование** (если не `--no-web`): `WebSearch` по запросам вида `"{исход} best practices 2026"`, `"AI/{домен} {исход} implementation patterns"`. Извлечь: тренды, как это решают лидеры, новые подходы. При rate-limit/ошибке — деградация на анализ только по `COMPETITORS.md`, отметить «без свежего ресёрча».
- **Анализ конкурентов** (из `COMPETITORS.md`): кто это закрывает и как; что работает; в чём мы можем иначе/лучше; усиливает ли это нашу дифференциацию.
- **Фильтр принципов** — проверить против **принципов из `VISION.md` продукта** (а не зашитых). Для каждого принципа: нарушает / соблюдает. Если возможность проваливает фильтр → пометить **SKIP** с обоснованием (в отчёт, раздел SKIPPED), в OPP не выносить.

### 6. Присвоить OPP-NNN
Выжившим возможностям дать стабильные id `OPP-001`, `OPP-002`, … (своё пространство нумерации, только внутри `vision-audit.md`; continue-from-max, если файл уже существует — **никогда не перенумеровывать**). `OPP` — это ещё **не** бэклог.

### 7. Записать отчёт
Заполнить `product/vision/vision-audit.md` по `reference/vision-audit.template.md`:
- **Summary** — счётчики Part A (🟢/🟡/🔴), Part B (achieved/avoided/opportunities), число OPP.
- **Part A** — R/Y/G по пунктам видения.
- **Part B** — таблица покрытия SDLC-исходов.
- **OPP-NNN** — по каждой: адресуемый исход, разрыв, находки ресёрча (тренд/конкуренты/дифференциация), проверка принципов, vision-fit, приоритет HIGH/MED/LOW.
- **SKIPPED** — возможности, провалившие фильтр принципов, с причинами.
- **Конкурентное позиционирование** — как OPP усиливают дифференциацию.
- **Рекомендуемый порядок** — приоритезированный список OPP (это и есть «roadmap» — раздел, не отдельный файл).
- **Таблица OPP→IMP** — пустая на этом этапе (заполняется `vision-to-backlog`).

### 8. Report
Напечатать краткое summary (счётчики Part A/B, число OPP по приоритетам) и путь к `vision-audit.md`. Завершить: **Next:** `/vision-to-backlog` — превратить OPP в IMP в бэклоге.

## Notes
- Доказательная база — карта системы, а не исходники: аудит стратегический, а не code review.
- В отчёте всегда помечать происхождение видения: «авторское» (полный вес оценок) vs «выведено и подтверждено» (`--lite`, меньший вес).
- `OPP` ≠ `IMP`: возможность — это анализ до бэклога; в бэклог её переводит только `vision-to-backlog`.
- Слишком много возможностей (>10): жёстко приоритезировать фильтром принципов; полные карточки OPP только для HIGH, остальное — списком.
- Read-only на исходники сервисов; пишет только в `product/vision/`.
