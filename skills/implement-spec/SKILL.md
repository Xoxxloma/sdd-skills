# Навык: реализация фичи по спецификации (SDD)

Цель — рабочий код без отсебятины и галлюцинаций. Ты не «пишешь как лучше»,
ты исполняешь спецификацию. Если спецификации ещё нет — это не твой навык,
сначала примени `spec-feature`.

## Алгоритм
1. **Прочитай контекст в этом порядке:** `GIGACODE.md` → `.gigacode/rules/react-ts.md`
   → `specs/<feature>.requirements.md` → `.gigacode/plans/<feature>.plan.md`.
2. **Возьми одну задачу.** Первую со статусом `todo`, у которой выполнены зависимости.
   Не бери несколько сразу. Если задача указана в запросе — бери только её. Поставь ей
   статус `doing`.
3. **Externalize reasoning (модель non-thinking).** Напиши явным текстом ПЕРЕД кодом:
   какие R-id закрываешь, какое наблюдаемое поведение должно появиться, какие файлы
   тронешь. Если требование неоднозначно — задай вопрос и СТОП, не угадывай.
4. **Размести логику правильно, потом тест, потом код.** Поведение задачи (дебаунс,
   гонки, валидация, переходы состояний, навигация) реализуй в хуке `useXxx` или чистой
   функции в `logic.ts`, а не в теле компонента. Компонент — тонкая оболочка.
   - На каждый ПОВЕДЕНЧЕСКИЙ R-id напиши падающий тест хука/функции
     (`renderHook`/прямой вызов, fake timers где нужно), в начале — комментарий с R-id.
     Затем минимальный код, чтобы тест позеленел.
   - ВИЗУАЛЬНЫЕ/разметочные R-id (рендер списка, спиннер, текст ошибки) тестом не
     закрывай — вынеси их в раздел «проверить вручную» в финальном ответе.
   - Если ловишь себя на тяжёлом мокинге (мок > 1–2 зависимостей, монтаж пол-приложения)
     — СТОП: логика не там. Вынеси её в хук/функцию и тестируй изолированно.
5. **Цикл самокоррекции (твоя сильная сторона).** Прогони гейты так, как описано в
   `GIGACODE.md` (типы — `npx tsc --noEmit`; линт/сборка/тесты — командами из
   `package.json`, менеджер пакетов — по lock-файлу). Команды не хардкодь, бери из проекта.
   - Зелёное → переходи к шагу 6.
   - Красное → вставь ПОЛНЫЙ вывод ошибки (tsc/eslint/vitest) обратно в рассуждение,
     локализуй причину, внеси ТОЧЕЧНУЮ правку. Не переписывай файл с нуля, не удаляй
     уже зелёные тесты, не отключай линт и не глуши типы ради прохождения. Повтори гейты.
   - **Предохранитель:** если после 3 итераций по одной задаче гейты всё ещё красные —
     остановись и опиши, что именно застряло (ошибка, гипотеза, где затык). Не накручивай
     бесконечные попытки.
6. **Обнови план:** статус задачи `doing → done`, при необходимости короткая заметка
   о принятом решении в журнал плана.
7. **Остановись и покажи диф.** Не начинай следующую задачу без подтверждения.

## Жёсткие запреты
- Не реализовывать требования, которых нет в `*.requirements.md`. Нашёл пробел — допиши
  спецификацию (или предложи), не «додумывай» в коде.
- Не добавлять зависимости, не трогать файлы вне задачи, не рефакторить «заодно».
- Не отключать правила линтера и не глушить типы, чтобы тесты прошли.
- Не коммитить в основную ветку и не пушить без явной просьбы.

## Формат ответа
1. Какую задачу взял (T-id) и какие R-id закрывает.
2. Список созданных/изменённых файлов (отметь, где хук/логика, где компонент).
3. Результат гейтов (какие команды запускал + вывод типов/линта/сборки/тестов логики).
4. **Проверить вручную:** список визуальных R-id для человека (что открыть и что увидеть).
5. Что осталось по плану (следующая задача).
