# Навык: реализация фичи по спецификации (SDD)

Цель — рабочий код без отсебятины и галлюцинаций. Ты не «пишешь как лучше»,
ты исполняешь спецификацию. Если спецификации ещё нет — это не твой навык,
сначала примени `spec-feature`.

## Где артефакты и где код
- **Артефакты** (`features/F-N/feature.md`, `requirements.md`, `plan.md`) лежат в текущем
  product-воркспейсе (CWD).
- **Код** пишется в **анализируемую систему**, которая находится в другом месте. **Пути к
  файлам бери ПРЯМО из `features/F-N/plan.md`** — план самодостаточен (колонка «файл(ы)» +
  `targetRoot` в его frontmatter; их проставил `spec-feature`). Манифест для записи кода
  больше не обязателен. Порядок определения пути:
  1. **Из плана** — пути задач + `targetRoot` из frontmatter `plan.md`. Это основной источник.
  2. **Fallback:** если в плане пути не указаны — `system/manifest.yaml` (`targetRoot` +
     `services[].path`), если он есть.
  3. Если ни плана-с-путями, ни манифеста нет (или путь не существует на диске) — **спроси
     человека** через `ask_user_question` (`header` ≤ 12; никакой записи кода до ответа), не
     угадывай путь.
  **Никогда не пиши код в CWD.**

## Алгоритм
1. **Прочитай контекст в этом порядке:** `GIGACODE.md` (если нет — `GIGACODE_BASE.md`) →
   `rules/react-ts.md` (если нет — `.gigacode/rules/react-ts.md`) → база знаний `system/`
   (**если есть** — для контекста; не обязательна) → `features/F-N/requirements.md` →
   `features/F-N/plan.md` (откуда берёшь пути к файлам — см. «Где артефакты и где код»).
2. **Возьми одну задачу.** Первую со статусом `todo`, у которой выполнены зависимости.
   Не бери несколько сразу. Если задача указана в запросе (T-n) — бери только её. Поставь ей
   статус `doing` в `features/F-N/plan.md`.
3. **Externalize reasoning (модель non-thinking).** Напиши явным текстом ПЕРЕД кодом:
   какие R-id закрываешь, какое наблюдаемое поведение должно появиться, какой сервис/папку
   (из манифеста) и какие файлы тронешь. Если требование неоднозначно — действуй по
   **протоколу B2** (`GIGACODE.md`/`GIGACODE_BASE.md`): спроси через **`ask_user_question`**
   (`header` ≤ 12) и **не пиши и не меняй код, пока человек не ответил** (в режиме «Спрашивать
   перед изменениями» confirmation-гейт перед файлом — вторая страховка). Не угадывай. Пробел
   в самих требованиях не закрывай кодом — см. «Жёсткие запреты».
4. **Размести логику правильно, потом тест, потом код.** Поведение задачи (дебаунс,
   гонки, валидация, переходы состояний, навигация) реализуй в хуке `useXxx` или чистой
   функции в `logic.ts`, а не в теле компонента. Компонент — тонкая оболочка.
   - На каждый ПОВЕДЕНЧЕСКИЙ R-id напиши падающий тест хука/функции
     (`renderHook`/прямой вызов, fake timers где нужно), в начале — комментарий с R-id.
     Затем минимальный код, чтобы тест позеленел.
   - ВИЗУАЛЬНЫЕ/разметочные R-id (рендер списка, спиннер, текст ошибки) тестом не
     закрывай — вынеси их в раздел «проверить вручную» в финальном ответе.
   - Если ловишь себя на тяжёлом мокинге (мок > 1–2 зависимостей, монтаж пол-приложения)
     — СТОП: логика не там. Вынеси её в хук/функцию и тестируй изолированно.
5. **Цикл самокоррекции (твоя сильная сторона).** Прогони гейты так, как описано в
   `GIGACODE.md` (типы — `npx tsc --noEmit`; линт/сборка/тесты — командами из
   `package.json`, менеджер пакетов — по lock-файлу) **в каталоге целевого сервиса**.
   Команды не хардкодь, бери из проекта.
   - Зелёное → переходи к шагу 6.
   - Красное → вставь ПОЛНЫЙ вывод ошибки (tsc/eslint/vitest) обратно в рассуждение,
     локализуй причину, внеси ТОЧЕЧНУЮ правку. Не переписывай файл с нуля, не удаляй
     уже зелёные тесты, не отключай линт и не глуши типы ради прохождения. Повтори гейты.
   - **Предохранитель:** если после 3 итераций по одной задаче гейты всё ещё красные —
     остановись и опиши, что именно застряло (ошибка, гипотеза, где затык). Не накручивай
     бесконечные попытки.
6. **Обнови план:** статус задачи `doing → done` в `features/F-N/plan.md`, при необходимости
   короткая заметка о принятом решении в журнал плана.
7. **Остановись и покажи диф.** Не начинай следующую задачу без подтверждения.

## Жёсткие запреты
- **ЗАПРЕЩЕНО писать или менять код, пока есть открытый уточняющий вопрос по требованию и
  человек на него не ответил** (протокол B2: спроси через `ask_user_question`, `header` ≤ 12 →
  правка только после ответа).
- Не реализовывать требования, которых нет в `features/F-N/requirements.md`. Нашёл пробел —
  допиши спецификацию (или предложи) и вернись в `spec-feature`, не «додумывай» в коде.
- Не писать код в product-воркспейс (CWD) — только по путям из `features/F-N/plan.md`
  (fallback — манифест; если ни того ни другого нет — спросить путь, не угадывать).
- Не добавлять зависимости, не трогать файлы вне задачи, не рефакторить «заодно».
- Не отключать правила линтера и не глушить типы, чтобы тесты прошли.
- Не коммитить в основную ветку и не пушить без явной просьбы.

## Формат ответа
1. Какую задачу взял (T-id), какие R-id закрывает, в какой сервис/папку (из манифеста) пишешь.
2. Список созданных/изменённых файлов (отметь, где хук/логика, где компонент).
3. Результат гейтов (какие команды запускал + вывод типов/линта/сборки/тестов логики).
4. **Проверить вручную:** список визуальных R-id для человека (что открыть и что увидеть).
5. Что осталось по плану (следующая задача).
