# Навык: спецификация фичи перед кодом (SDD, фаза Spec → Plan)

Модель non-thinking: она не «додумает» неоднозначный бриф, а быстро закоммитится в
первое допущение. Поэтому твоя работа здесь — вынести всё мышление в артефакты, которые
потом исполнит навык `implement-feature`. Чем точнее спека, тем меньше галлюцинаций.

## Inputs / outputs
- Читает контекст: `GIGACODE.md` (если нет — `GIGACODE_BASE.md`) → `rules/react-ts.md`
  (если нет — `.gigacode/rules/react-ts.md`) → **базу знаний `system/`** (`system-map.md`,
  карточки сервисов, `manifest.yaml`). Если `system/` нет → стоп: «сначала `/system-map <path>`».
- Источник фичи — по аргументу (см. «Резолв входа»).
- Пишет папку фичи `features/F-N/`:
  - `feature.md` — БТ (канонично с этого момента);
  - `requirements.md` — EARS-требования (R-1, R-2, …);
  - `plan.md` — атомарный план задач (T-1, T-2, …).

## Numbering
Единое пространство `F-N` с `backlog/` и `features/`. При промоушене из бэклога **номер
сохраняется** (`backlog/F-7.feature.md` → `features/F-7/`), новый не выделять. Для свежего
брифа — `max(N)+1` по сканированию `backlog/*.feature.md` и `features/F-*/`. Не перенумеровывать.

## Резолв входа
- **Номер `F-N`** → прочитать `backlog/F-N.feature.md` (или уже `features/F-N/feature.md`,
  если папка есть). Это БТ-источник. После создания папки пометить исходный
  `backlog/F-N.feature.md` как `status: promoted` (канон теперь в `features/F-N/`).
- **Путь до `feature.md`** → прочитать его как БТ-источник; номер — из его frontmatter `id`.
- **Человеческий бриф** (произвольный текст) → выделить следующий `F-N`, БТ нет в бэклоге;
  пишем **только** в `features/F-N/` (запись в backlog не создаём).

## Алгоритм
1. **Контекст стека и системы:** прочитай файлы из «Inputs» (GIGACODE → rules → `system/`).
   Затем по «Резолв входа» определи `F-N` и источник фичи.
2. **Прими бриф/БТ.** Найди пробелы по осям ниже и, если их нет, задай максимум 3–5
   уточняющих вопросов и **СТОП** (жди ответов, не выдумывай):
   - источник данных и форма (пропс/хук, формат ответа, типы);
   - состояния: загрузка, пусто, ошибка, граничные значения;
   - взаимодействие: клавиатура, фокус, валидация ввода;
   - доступность (роли/ARIA для интерактивных виджетов);
   - какие сервисы затронуты (сверь с `system/manifest.yaml`);
   - что явно вне области фичи.
3. **Положи БТ в папку фичи:** запиши `features/F-N/feature.md` — это исходный БТ
   (скопированный из бэклога/файла, либо сгенерированный из брифа в формате
   `propose-improvements` БТ). Заполни/уточни frontmatter `services:` из манифеста.
4. **Сгенерируй `features/F-N/requirements.md`** в формате EARS:
   - frontmatter: `id: F-N`, `source: feature.md`, `services: [...]` (из манифеста);
   - атомарные требования R-1, R-2, … — каждое проверяемо одним-двумя тестами;
   - покрой ОБЯЗАТЕЛЬНО: нормальный путь, загрузку, пустой результат, ошибку,
     граничные значения, клавиатуру/фокус для интерактива;
   - помечай каждое требование как поведенческое (тестируемо в хуке/функции) или
     визуальное (проверяется глазами) — это нужно плану для столбца «проверка»;
   - ключевые слова: `СИСТЕМА ДОЛЖНА` (всегда), `ПОКА … СИСТЕМА ДОЛЖНА` (состояние),
     `КОГДА … СИСТЕМА ДОЛЖНА` (событие), `ЕСЛИ … ТО СИСТЕМА ДОЛЖНА` (нежелательное);
   - добавь 2–3 примера приёмки Дано/Когда/Тогда на самые рискованные R (гонки,
     ошибки, дебаунс и т.п.).
5. **Сгенерируй `features/F-N/plan.md`**:
   - frontmatter: `id: F-N`, `source: requirements.md`, `services: [...]` (из манифеста —
     `implement-feature` по нему найдёт, куда писать код);
   - таблица задач T-1, T-2, … со столбцами: задача, закрывает (R-id), проверка, зависит от, статус;
   - **группируй поведение в отдельные хук/util-задачи**, чтобы его можно было тестировать
     изолированно; рендер и проводка ARIA — отдельными задачами поверх готовых хуков;
   - столбец «проверка»: `[тест]` для логических задач (хук/функция) или `[ручками]` для
     рендера/визуала;
   - каждая задача атомарна и закрывается за одну сессию;
   - первая задача обычно «каркас + типы + доступная разметка» `[ручками]`,
     поведенческие — `[тест]`;
   - добавь раздел «Чек-лист ручной приёмки» (пункты по визуальным R-id: что открыть,
     что увидеть) и пустой «Журнал решений».
6. **Покажи все три файла и СТОП.** Жди ревью человека. Код не пиши — это сделает
   `implement-feature` по утверждённому плану. Завершить: **Next:** `/implement-feature F-N`.

## Жёсткие запреты
- Не писать код и не создавать компоненты на этом шаге.
- Никаких «пишите хороший код» — только конкретные проверяемые требования.
- Не раздувать: одна фича — один `features/F-N/`. Если фича огромная — предложи разбить на
  несколько фич, но не специфицируй «про запас».
- Read-only на анализируемую систему; пишет только в workspace (CWD).
